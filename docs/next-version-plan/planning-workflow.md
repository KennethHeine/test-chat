# Planning Workflow

## Goal
Define a repeatable flow that converts an idea into a GitHub-executable delivery plan.

## Phase 1: Define the goal
The system should help the user define:
- long-term goal
- business value
- problem statement
- target outcome
- success criteria
- assumptions
- constraints
- risks

## Phase 2: Define required research
Before coding starts, the system should identify and document:
- domain unknowns
- architecture unknowns
- security and compliance questions
- infrastructure requirements
- integration dependencies
- data model questions
- operational concerns
- user experience decisions

Output:
- research checklist
- open questions
- decision log
- dependency map

## Phase 3: Create milestone plan
The system should break the long-term goal into milestones.

Each milestone should include:
- milestone name
- milestone goal
- scope
- dependencies
- acceptance criteria
- exit criteria
- estimated issue list

## Phase 4: Create GitHub execution structure
For each milestone, the system should prepare:
- GitHub project items if needed
- milestone record
- branch naming convention
- issue backlog
- issue ordering
- labels
- workflow triggers

## Phase 5: Prepare issue quality
Each issue should contain:
- title
- purpose
- problem to solve
- expected outcome
- scope boundaries
- technical context
- dependencies
- acceptance criteria
- testing expectations
- links to research and decisions

## Phase 6: Orchestrate issue execution
The system should execute milestone issues one by one through GitHub workflows and coding agent flows.

## Principle
No milestone should enter autonomous execution until planning and research quality are high enough for issues to be implementation-ready.
